## Android‑додаток «Малин транспорт» (як Jakdojade)

Цей документ описує цілі, фічі та архітектурний інтерфейс майбутнього Android‑додатку для локального транспорту Малина, натхненного застосунком Jakdojade (Kraków / PL).

---

## 1. Цілі продукту

- **Оффлайн‑орієнтований розклад**: швидкий доступ до розкладу місцевого транспорту навіть без стабільного інтернету (дані періодично оновлюються з бекенду).
- **Просте планування поїздки «З → До»** в межах існуючих маршрутів, з розрахунком часу в дорозі на базі сегментів між зупинками.
- **Інтерактивна карта**: візуалізація маршруту, зупинок, та відрізку між «З» та «До».
- **Локальний фокус**: працює лише з маршрутами Малина (дані `malyn_transport.json`, `stops_coords.json`).

---

## 2. Функціональні фічі

### 2.1. Фічі, які вже сьогодні реалізовані на сайті (і мають бути в апці)

- **Список маршрутів**
  - Перелік усіх доступних маршрутів з номерами та парою кінцевих: `from` ↔ `to`.
  - Кількість рейсів по кожному маршруту.

- **Фільтрація за зупинкою та маршрутом**
  - Пошук зупинки через комбобокс (автокомпліт).
  - Фільтр «Усі зупинки» / конкретна зупинка.
  - Фільтр «Усі маршрути» / конкретний маршрут.

- **Детальна сторінка маршруту**
  - Вибраний маршрут з напрямком:
    - `there`: `from` → `to`.
    - `back`: `to` → `from`.
  - Кнопки перемикання напрямку (туди / назад).
  - Відображення вартості проїзду (якщо є `supplement.fare`).

- **Вибір «З» та «До» всередині маршруту**
  - Випадаючі комбобокси для зупинок `fromStop` (З) та `toStop` (До) з урахуванням напрямку та доступності зупинок.
  - Автоматична корекція напрямку при виборі пар зупинок (логіка `getImpliedDirection`).

- **Розклад поїздок**
  - Таблиця `Відправлення (З)` / `Прибуття (До)` з урахуванням:
    - реальних сегментів між зупинками (`segmentDurations.json`, `getDurationFromStartSec`, `getSegmentDurationSec`) для перевірених маршрутів;
    - дефолтного часу між зупинками (`getMinsBetweenStops`) для інших маршрутів.
  - Альтернативний режим: списки часів лише по напрямках, якщо З/До не задані.
  - Автовибір найближчого рейсу від поточного часу (часова зона `Europe/Kyiv`) з урахуванням напрямку (`findNearestTrip`, `getKyivMinutesNow`).

- **Карта маршруту**
  - Підвантаження координат з `stops_coords.json`.
  - Маркери всіх зупинок маршруту в конкретному напрямку.
  - Лінія маршруту (перевірені маршрути з `VERIFIED_ROUTE_IDS`).
  - Виділена товстою лінією ділянка між З та До.
  - Автоматичне підлаштування зуму під весь маршрут або ділянку З → До.

- **Список зупинок уздовж маршруту**
  - Відображення зупинок в обраному напрямку з:
    - розрахунковим часом прибуття до кожної зупинки;
    - тривалістю між поточною та наступною зупинкою («X хв Y сек» або «Y сек»);
    - візуальним виділенням сегменту між З та До на вертикальній шкалі.

- **Пошук найближчих зупинок за геолокацією**
  - Використання `navigator.geolocation` для визначення позиції.
  - Обчислення відстані до кожної зупинки (`haversineDistance`).
  - Відображення топ‑5 найближчих зупинок з відстанню в метрах/кілометрах.

- **QR‑посилання з зупинок**
  - Функція `buildStopRouteQrUrl(routeId, stopName, direction)` для відкриття сторінки маршруту із зафіксованими зупинкою та напрямком (для апки — deep‑link схема).

### 2.2. Фічі з Jakdojade, які варто закласти архітектурно (навіть якщо реалізуємо пізніше)

- **Планування маршруту «будь‑яка точка → будь‑яка точка»**
  - Вибір не лише зупинок, а й довільної точки на карті / геолокації як початку/кінця.
  - Побудова ланцюжка: пішки до найближчої зупинки → маршрут(и) → пішки від останньої зупинки.

- **Планування по часу**
  - Варіанти:
    - «Вирушити зараз».
    - «Вирушити не раніше ніж о HH:MM».
    - «Прибути не пізніше ніж о HH:MM».
  - Обчислення оптимального варіанту поїздки за заданим критерієм (мінімальний час у дорозі, мінімальна кількість пересадок тощо).

- **Улюблені зупинки / маршрути**
  - Збереження зупинок та маршрутів у «Вибране» для швидкого доступу.
  - Опціонально — «Улюблені поїздки» (конкретна пара З/До на певному маршруті).

- **Історія пошуків**
  - Короткий список останніх запитів (З/До, час).

- **Push‑нагадування та сповіщення**
  - Оповіщення «час виходити» на поїздку.
  - В майбутньому — сповіщення про затримки/зміни (коли будуть відповідні дані з бекенду).

- **Покращена робота з картою**
  - Показ усіх зупинок міста на мапі з кластеризацією.
  - Перемикання шару «маршрути» / «зупинки».

- **Мультимовність**
  - Принаймні українська / англійська, можливо польська.

- **Оновлення даних**
  - Перевірка наявності нових розкладів/координат.
  - Керування кешем (розмір, дата оновлення).

## 3. Архітектурний інтерфейс (доменна модель та шари)

### 3.1. Джерела даних (Data Sources)

Базуємось на поточних структурах у фронтенді:

- **Файли даних (у бекенді / CDN):**
  - `malyn_transport.json` (дані з data.gov.ua + supplement):
    - `records`: сирі поїздки (аналог `TransportRecord`).
    - `supplement.routes`: додаткові метадані по маршрутах (from/to, тощо).
    - `supplement.stops.stops_by_route`: перелік зупинок по маршруту.
    - `supplement.fare`: базова вартість.
  - `stops_coords.json`:
    - `center`: центральна точка мапи.
    - `stops`: координати зупинок.
  - `segmentDurations.json`:
    - `defaultSec` та `segments["routeId|stopFrom|stopTo"]`.

В Android‑додатку це можна моделювати як:

- **RemoteDataSource**
  - `suspend fun fetchTransportData(): TransportDataDto`
  - `suspend fun fetchStopsCoords(): StopsCoordsDto`
  - `suspend fun fetchSegmentDurations(): SegmentDurationsDto`

- **LocalDataSource (кеш/БД)**
  - Room / SQLDelight / інша БД:
    - таблиці `routes`, `stops`, `trips`, `segments`, `favorites`.
  - Методи оновлення з DTO та читання для домену.

### 3.2. Доменна модель (Domain Layer)

Має відповідати поточним типам у `LocalTransportPage`:

- **Route**
  - `id: String`
  - `from: String?`
  - `to: String?`
  - `trips: List<Trip>`
  - `supplement: RouteSupplement?`

- **Trip**
  - `id: String`
  - `routeId: String`
  - `direction: Direction` (`THERE` / `BACK`)
  - `baseTimeMinutes: Int` (аналог `block_id` після парсингу)

- **Stop**
  - `name: String`
  - `lat: Double`
  - `lng: Double`

- **RouteStopOrder**
  - `routeId: String`
  - `stopName: String`
  - `orderThere: Int?`
  - `orderBack: Int?`
  - `belongsTo: BelongsTo` (`THERE`, `BACK`, `BOTH`)

- **SegmentDuration**
  - `routeId: String`
  - `fromStop: String`
  - `toStop: String`
  - `durationSec: Int`

- **Fare**
  - `amount: Int`
  - `currency: String` (поки `UAH`, але варто дозволити розширення).

### 3.3. Use‑cases (Application Layer)

Приблизний набір use‑cases (будемо реалізовувати по черзі):

- **LoadLocalTransportData**
  - Скачати/оновити `TransportData`, `StopsCoords`, `SegmentDurations`.
  - Синхронізувати з локальною БД.

- **GetRoutesList**
  - Повернути список маршрутів з фільтрами:
    - за зупинкою (як у `routeHasStop`);
    - за номером маршруту.

- **GetRouteDetails**
  - Повернути:
    - маршрут;
    - список зупинок у напрямку;
    - сегменти з тривалістю;
    - часи рейсів.

- **GetNearestTripForRoute**
  - На вхід: `routeId`, `direction?`, `nowMinutes`, опціонально `fromStop`/`toStop`.
  - На вихід: найближчий `Trip` + розклад по З/До.

- **GetNearestStopsByLocation**
  - Вхід: `lat`, `lng`.
  - На вихід: топ‑N зупинок з відстанню.

- **PlanTripBetweenStops (майбутня фіча)**
  - Вхід: `fromStop`, `toStop`, `desiredTime`, `mode (DEPART_AFTER / ARRIVE_BEFORE)`.
  - На вихід: один або кілька варіантів поїздки (можливо з пересадками в майбутньому).

- **ManageFavorites**
  - `AddFavoriteStop`, `RemoveFavoriteStop`, `GetFavoriteStops`.
  - `AddFavoriteRoute`, `RemoveFavoriteRoute`, `GetFavoriteRoutes`.

## 4. Екрани Android‑додатку

### 4.1. Головний екран

- Блок **«Знайти маршрут»**:
  - Поле «Звідки» (зупинка / геолокація / улюблені).
  - Поле «Куди».
  - Кнопка «Знайти» → відкриває екран результатів.
- Блок **«Найближча зупинка»**:
  - Кнопка «📍 Знайти найближчу» (аналог `handleFindNearest`).
  - Список топ‑N зупинок з відстанями, тап по елементу → розклад по цій зупинці.
- Блок **«Маршрути»**:
  - Кнопка переходу до повного списку маршрутів (екран 4.2).

### 4.2. Екран «Маршрути»

- Список маршрутів (`GetRoutesList`):
  - Номер, напрямок (`from` ↔ `to`), к‑сть рейсів.
  - Пошук/фільтрація за номером.
  - Опція фільтра за зупинкою (модальне вікно / окремий фільтр).
  - Тап по маршруту → екран «Деталі маршруту».

### 4.3. Екран «Деталі маршруту» (аналог `LocalTransportPage` з routeId)

- Верхня частина:
  - Номер маршруту, напрямок (`there` / `back`).
  - Кнопки перемикання напрямку.
  - Вартість проїзду.

- Блок вибору З/До:
  - Два комбобокси, стрілка «⇄» для зміни напрямку та swap З/До.

- Блок розкладу:
  - Таблиця `Відправлення` / `Прибуття` для вибраних З/До.
  - Або списки часів по напрямках, якщо З/До не вибрані.

- Блок карти:
  - Маршрут на мапі, виділений сегмент З → До.
  - Тап по зупинці на карті → підсвічування в списку зупинок.

- Блок зупинок уздовж маршруту:
  - Ліст з часами прибуття, тривалістю до наступної зупинки, маркерами «З» / «До».

### 4.4. Екран «Результати пошуку поїздки» (майбутня фіча)

- Список варіантів:
  - Для кожного: час відправлення, час прибуття, загальна тривалість, кількість пересадок.
  - Тап → детальна розкладка по етапах (пішки / маршрут / пересадка).

### 4.5. Екран «Улюблене»

- Вкладка «Зупинки» та «Маршрути».
- Швидкий перехід до розкладу/деталей.

## 5. Синхронізація з існуючим фронтендом

- **Єдине джерело істини даних**:
  - Android‑додаток використовує ті самі JSON (`malyn_transport.json`, `stops_coords.json`, `segmentDurations.json`), що й фронтенд.
  - Для зручності варто:
    - зафіксувати стабільні схеми DTO в бекенді;
    - документувати зміни (changelog для форматів).

- **Спільна логіка часу/сегментів**:
  - Алгоритмічну частину (розрахунок тривалостей, робота з сегментами) вже реалізовано в скриптах та `routeTiming.ts`.
  - Для Android варто:
    - або продублювати ці алгоритми на Kotlin;
    - або перенести частину логіки в бекенд (API, яке повертає вже готові тривалості / розклади).

- **Deep‑link’и / QR‑коди**
  - Те, що зараз генерується як URL (`/localtransport/route/:id?stop=...&to=...&time=...&dir=...`), в додатку має мати еквівалент:
    - схема `malyntransport://route/:id?stop=...&to=...&time=...&dir=...`.

## 6. Порядок реалізації

1. **MVP (на базі поточного функціоналу сайту)**
   - Локальне збереження `malyn_transport.json`, `stops_coords.json`, `segmentDurations.json`.
   - Список маршрутів, екран деталей маршруту з вибором З/До, розкладом та картою.
   - Пошук найближчої зупинки.
2. **Розширення можливостей**
   - Улюблені зупинки/маршрути.
   - Базове планування поїздки між зупинками.
3. **Фічі рівня Jakdojade**
   - Планування «точка → точка» з пішохідними сегментами.
   - Продвинутий пошук по часу.
   - Історія, сповіщення, більш гнучка робота з картою.

Цей документ можна поповнювати по мірі того, як будемо додавати нові вимоги чи змінювати існуючі структури даних.
